# Skill 调用准确率提升：现状、思路与方案
 
**本文定位**：把「为什么要做、做什么、怎么做」写成一份共识文档；战术细节（场景库字段、本地 Chroma、三层路由展开等）见同目录 **[hermes-skill-routing-from-scenarios.md](./hermes-skill-routing-from-scenarios.md)**。

---

## 1. 现状

### 1.1 我们有什么

- **已定义的技能（Skill）**：数量可大可小；默认假设 **`name` 与 `description`（及整体 SKILL 内容）已完备且合理**，无需靠外挂去「补写技能本身」。
- **评测集**：格式为 **问题 → 应调用的 skill（可多选）**；默认假设 **标注即 ground truth**，用于衡量「是否调对了 skill」。

### 1.2 Hermes 原生行为（约束与痛点）

在 **不改动 Hermes 内核、不替换背后 LLM** 的前提下，框架侧典型的 skill 暴露方式是：

- 系统提示中提供技能 **索引**（如 `name` + 简短 `description`）；
- 由 **模型自行决定** 是否调用 `skill_view`、以及传入的 **`name` / `file_path` 等参数**。

因此：

- **准确率高度依赖 LLM** 在长列表中的选择与遵循能力；
- **难以在中间层干预**「模型最终选了哪一个 skill」——除非在**进入 Hermes 的上下文**上做文章（额外指令、路由层、网关配置等）。

### 1.3 核心矛盾（一句话）

**技能越多、场景越细，仅靠索引里的 name/description 让模型稳定对齐「评测集期望的调用」，方差大、可复现弱。**

---

## 2. 思路

### 2.1 目标表述

在 **不改模型权重、不重写 SKILL 元数据假设** 的前提下，提升 **「用户问题 → 实际 `skill_view` 调用」** 与 **评测集标注** 的一致性。

### 2.2 核心判断

- **评测集**定义的是我们认可的 **行为标准**（哪些问题该触达哪些 skill）。
- **Hermes 原生路径**缺少「从用户话到精确 `skill_view` 字符串」的 **可编程、可检索、可迭代** 的中间层。
- **外挂（编排层）** 可以在用户问题进入主对话前，注入 **与当前问题强相关的、带明确工具参数的短提示**（例如 `[Routing hints]`），使 **同一 LLM** 在后续步骤中 **更稳定地** 执行与评测集一致的 skill 加载顺序。

换言之：**不是去「训练模型记住评测集」，而是用上下文把「评测集里蕴含的调用决策」以可检索的形式喂给 agent，使实际行动拟合预期——拟合的是 tool 调用行为，不是模型内部表示。**

### 2.3 与 `hermes-skill-routing-from-scenarios.md` 的关系

该文档描述的是 **同一思路的展开版**：场景资产化、三层路由（规则 / RAG / 索引兜底）、索引降噪、闭环迭代，以及 **本地快速 RAG 调测** 的工程选项。本文不重复所有操作细节，只保留 **决策级归纳**。

---

## 3. 解决方案（分层落地）

建议按 **可控性从高到低** 组合使用；评测集既用于 **离线验收入库质量**，也用于 **端到端回归**。

### 3.1 层 A：硬路由（强对齐、低方差）

适用于 **入口确定** 的场景（频道、项目、客户、固定产品形态等）：

- Hermes 网关 **`channel_skill_bindings`**（按 channel/thread 绑定 skill 列表）；
- 或 **编排层规则表**：关键词 / 正则 / 轻量分类器 → 直接注入「必须先 `skill_view(name='…')`」的指令块。

**价值**：与评测集中「固定场景」高度重合时，**几乎不依赖检索**即可对齐。

### 3.2 层 B：场景 RAG（长尾、可迭代）

将 **真实问题表述**（及评测集中的问题变体）写入 **场景库**，每条关联：

- **`skill_view` 可用的精确字符串**（及可选 `file_path`）；
- 必要时 **negative**（易混 skill 显式排除）。

**在线**：用户当前问题 → 检索 Top-K（建议 K 很小，如 3）→ 拼成 **`[Routing hints]`** → 置于 **首条用户消息前**（或等价 system/user 补充），再进入 Hermes。

**价值**：把「评测集里的问题 → skill 集合」变成 **可检索的监督信号**；迭代场景库即可提升覆盖率，而无需改模型。

### 3.3 层 C：索引与配置降噪（减少误选空间）

在 **不否定 name/description 已完备** 的前提下，仍可通过 **减少「同时出现在模型眼前的无关 skill」」** 降低误召：

- SKILL frontmatter 中 **`metadata.hermes`** 的条件字段（`requires_*` / `fallback_for_*`），使当前会话 **不该出现的 skill 不出现在系统索引中**；
- **`skills.disabled` / `platform_disabled`**、多 profile / 拆分 `external_dirs`，按环境或产品线 **缩小可见技能集**。

**价值**：与层 B 互补——RAG 负责「这句话像谁」，索引裁剪负责「别给模型看一整本黄页」。

### 3.4 验收与闭环

- **路由层指标**（可快速迭代）：同一评测问题，外挂输出的 skill 集合与 gold 的 **recall@K / set match** 等。
- **端到端指标**：Hermes 对话中是否实际出现预期的 `skill_view` 调用及参数。
- **错例回灌**：误召、漏召写入场景库（含 `negative` 与表述变体），持续缩小与评测集的差距。

---

## 4. 边界与诚实说明

1. **外挂不形成形式化保证**：模型仍可能忽略或与 hints 冲突；目标是 ** statistically 提升** 与评测集的一致性。
2. **RAG 覆盖依赖场景库与评测分布一致**：评测中的新问题若与库中表述差异大，需靠 **持续入库与混合检索（向量 + 关键词）** 补洞。
3. **一问多 skill**：hints 中需 **显式列出多个 `skill_view` 及建议顺序**（或分步说明），否则模型可能只加载其一。
